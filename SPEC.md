# SPEC.md — Ageless Strength

> 欧美市场品牌名：**Ageless Strength** | Strength Training for Every Age
> 内部开发代号：**SeniorStrength**（项目目录保留原名）

## 1. 产品概述

### 产品定位
一款专门为50+老年人设计的居家力量训练工具，通过AI姿态实时检测确保动作安全，用进度追踪保持训练动力，用家庭账号让子女参与监督，形成"预防肌少症"的商业闭环。

### 核心价值
- **老人端**：专属力量训练 + AI安全检测 + 进度可视化
- **子女端**：远程监督 + 鼓励互动 + 医疗报告
- **商业化**：订阅制（个人/家庭）+ 保险对接（后期）

### 目标市场
- 首发：北美/英语市场（美国为主）
- 次选：欧洲（德国、法国）、澳大利亚

## 2. 用户画像

### 老人端（直接用户）
| 维度 | 描述 |
|------|------|
| 年龄 | 55-85岁，以65-75岁为主力 |
| 健康状况 | 关注健康，体检有肌少症风险或已确诊 |
| 技术能力 | 能用微信/iMessage，会看视频，最好能独立使用iPad |
| 支付方式 | 子女帮助订阅为主，自己支付为辅 |

| 核心动机 | 延缓衰老、减少去医院、想陪孙子孙女玩 |

| 维度 | 描述 |
|------|------|
| 年龄 | 30-50岁，中青年 |
| 地域 | 与父母异地或同地 |
| 核心障碍 | 无法随时陪伴父母运动，不放心老人自己练 |
| 核心动机 | 希望父母健康、减少医疗负担、想让父母动起来 |
| 付费意愿 | 较强，愿意为"让父母安全锻炼"付费 |

## 3. 功能模块

### 3.1 老人端 APP

#### F1: 视频课程模块
- **课程分类**：入门（椅子训练）、基础（弹力带）、进阶（自重深蹲）
- **每节课**：10-20分钟，包含热身+正式训练+放松
- **视频要求**：老年演示者（55岁+）、大字幕、语速慢、动作分解清晰
- **交互**：播放/暂停、重新播放、上一节/下一节

#### F2: AI 姿态实时检测
- **触发**：手机/平板前置摄像头开启时自动运行
- **检测内容**：膝盖角度、手肘角度、脊柱是否弯曲、平衡感
- **实时反馈**：语音或文字提示"膝盖再弯曲一点"、"背挺直"
- **训练后报告**：动作质量分数（0-100）、改进建议

#### F3: 打卡与进度追踪
- **每日打卡**：完成训练后一键打卡
- **进度页面**：坚持天数、累计训练时长、肌肉进步曲线（估算）
- **成就系统**：7天连续、30天连续、100天等里程碑徽章

#### F4: 营养建议
- **蛋白质摄入提醒**：根据训练强度推送每日蛋白质目标
- **简单食谱推荐**：高蛋白家常菜（图文形式）
- **补水提醒**：训练后提醒喝水

#### F5: 医护报告
- **月度报告**：训练次数、累计时长、进步评估
- **PDF导出**：可分享给家庭医生
- **数据脱敏**：报告不包含个人隐私数据

### 3.2 子女端 APP

#### F1: 家庭绑定
- **扫码绑定**：老人端生成绑定码，子女扫码关联
- **多子女支持**：一个老人账号最多5个子女绑定

#### F2: 进度看板
- **本周训练**：次数/天数
- **坚持天数**：当前连续天数
- **最后训练**：时间（让子女知道父母有没有在练）
- **鼓励消息**：子女可发送表情或短句

#### F3: 订阅管理
- **家庭计划**：$14.99/月，可添加父母账号
- **续费提醒**：提前3天推送子女
- **账单历史**：Stripe对接

#### F4: 医生报告
- **查看权限**：可查看父母月度医疗报告
- **分享功能**：一键分享给家庭医生

### 3.3后台管理系统（运营方）

#### F1: 内容管理
- 课程视频上传、排序、启用/下线
- 课程难度标签管理

#### F2: 用户管理
- 用户列表、账号状态、订阅状态
- 家庭账号关系管理

#### F3: 数据分析
- DAU/WAU/MAU
- 课程完播率
- 打卡率
- 流失率

## 4. 核心用户流程

### 4.1 老人首次使用流程
```
下载APP → 手机号/邮箱注册 → 选择健身目标 →
拍摄老人照片（年龄估算） → 观看入门视频 → 
开启第一次训练 → AI检测开启提示 → 完成训练 →打卡 → 
显示坚持X天页面 → 邀请子女（可选）
```

### 4.2 子女绑定流程
```
下载APP → 注册 → 扫码/输入绑定码 → 绑定老人账号 → 
进入进度看板 → 看到父母训练数据 → 发送鼓励
```

### 4.3 订阅转化流程
```
免费体验（3节课程）→ 训练数据积累 → 弹出付费提示 →
选择个人/家庭计划 → Stripe支付 → 订阅成功 → 解锁全部课程
```

## 6. 技术方案

### 6.1 技术栈
| 层级 | 技术选型 | 说明 |
|------|---------|------|
| **APP框架** | Flutter 3.x | 跨平台（iOS/Android），一套代码 |
| **AI姿态检测** | MediaPipe (Google) | 开源，实时姿态检测，无需昂贵GPU |
| **后端** | Python FastAPI | 轻量级API，快速开发 |
| **数据库** | PostgreSQL | 用户数据、课程数据、进度数据 |
| **文件存储** | AWS S3 / Cloudflare R2 | 视频存储 |
| **订阅管理** | RevenueCat | 防流失、analytics、Stripe对接 |
| **支付** | Stripe | 订阅收款 |

### 6.2 数据模型（核心实体）
```
User (老人账号)
├── id, phone/email, age, gender
├── fitness_goal, health_notes
└── family_id → Family

Family (家庭)
├── id
└── members: [elder, adult1, adult2...]

Course (课程)
├── id, title, description, difficulty
├── video_url, thumbnail_url, duration
└── exercises: [Exercise]

TrainingSession (训练记录)
├── id, user_id, course_id
├── completed_at, duration, ai_score
└── exercises_completed: [bool]

Subscription
├── id, user_id, plan_type
├── status, current_period_end
└── stripe_subscription_id
```

### 6.3 API 核心接口
```
POST /auth/register          用户注册
POST /auth/login 用户登录

GET  /courses                课程列表
GET  /courses/{id}课程详情
POST /training/start 开始训练（记录）
POST /training/complete     完成训练（记录）

GET  /progress/weekly 周进度
GET  /progress/monthly      月度报告（PDF）

POST /family/bind           绑定家庭成员
GET  /family/members        家庭成员列表

POST /subscription/create 创建订阅（Stripe）
POST /webhook/stripe Stripe回调
```

## 8. 竞品分析

### 8.1 竞品概览
| 竞品 | 模式 | 优点 | 缺点 |
|------|------|------|------|
| SilverSneakers | 健身房+App | 保险覆盖、品牌强 | 依赖健身房、无AI、无居家 |
| YouTube老年健身 | 免费视频 | 用户多、内容丰富 | 无追踪、无体系、无反馈 |
| Nike Training Club | 通用健身App | 视频质量高 | 界面复杂，老人用不来 |
| 健身类博客 | YouTube | 内容有趣 | 无商业闭环、无个性化 |

### 8.2 我们的差异化
✅专门为老人设计的力量训练（非瑜伽/有氧）
✅ AI姿态实时检测（确保安全）
✅ 居家场景（不用去健身房）
✅ 家庭账号体系（子女参与）
✅ 医护报告（可给医生看）
✅ 订阅制商业闭环

## 10. 风险与应对

| 风险 | 等级 | 应对 |
|------|------|------|
|老人用户学习成本高 | 高 | 先做iPad适配，大屏+大字体，流程极简 |
| AI姿态检测精度不足 | 中 | MediaPipe调优，保守阈值，不给错误反馈 |
| 医学背书获取困难 | 中 | 先以"健身指导"定位，避免医疗声明 |
| Apple审核拒绝老年类App | 中 | 准备老年用户测试报告，说明无医疗内容 |
| 保险对接周期过长 | 低 | 先做订阅制，保险对接作为Phase 3后探索 |
